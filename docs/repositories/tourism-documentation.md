# Профиль репозитория tourism-documentation

## Назначение

`tourism-documentation` предназначен для долговременной архитектурной,
продуктовой и эксплуатационной документации платформы. После создания
private remote он клонируется рядом с `tourism-platform` как Git submodule
общего superproject.

## Ответственность

- Хранить обзор архитектуры и границы доменных модулей.
- Фиксировать архитектурные решения и причины изменений.
- Описывать публичные контракты, сценарии и словарь терминов.
- Поддерживать инструкции разработки, выпуска и эксплуатации.
- Связывать требования с проверяемыми результатами.
- Проверять актуальность, владельцев и даты пересмотра документов.

## Вне целей

- Дублирование исходного кода или автоматически создаваемых контрактов.
- Хранение секретов, персональных данных и закрытых ключей.
- Подмена трекера задач и системы оперативного мониторинга.
- Утверждение продукта в качестве официального государственного
  приложения.
- Создание remote repository этим документом.

## Будущая реализация и стек

- Markdown с правилами, совместимыми с Markdownlint.
- Диаграммы как текст там, где они упрощают понимание.
- Автоматическая проверка ссылок, стиля и структуры.
- Генерация сайта документации из проверенных исходников при
  необходимости.
- Версионирование материалов вместе с выпусками контрактов.
- Шаблоны решений, эксплуатационных инструкций и описаний доменов.

## Интеграции

- `tourism-platform` как главный репозиторий и источник общих соглашений.
- `tourism-backend` для серверных контрактов и эксплуатационных сведений.
- `tourism-mobile` для пользовательских потоков и требований к клиенту.
- `tourism-infrastructure` для схем сред и инструкций реагирования.
- Система CI для проверки ссылок и правил Markdown.

## Результаты

- Актуальный обзор системы и карта ответственности.
- Реестр архитектурных решений.
- Каталог публичных контрактов и правил совместимости.
- Инструкции для разработки, выпуска, отказов и восстановления.
- Словарь согласованных доменных терминов.
- Процесс проверки и архивирования устаревших материалов.

## Поэтапный план

### Этап 1. Основа

- [ ] Определить структуру, владельцев и шаблоны.
- [ ] Зафиксировать словарь и правила именования.
- [ ] Описать модульный монолит и допустимые зависимости.
- [ ] Добавить проверку Markdown и ссылок.

### Этап 2. Контракты

- [ ] Описать версии публичного программного интерфейса.
- [ ] Зафиксировать события и ошибки на концептуальном уровне.
- [ ] Добавить политику совместимости и устаревания.
- [ ] Связать контракты с автоматическими проверками.

### Этап 3. Эксплуатация

- [ ] Описать выпуск, миграции и откат.
- [ ] Добавить инструкции для типовых отказов.
- [ ] Зафиксировать резервное копирование и восстановление.
- [ ] Определить показатели качества документации.

### Этап 4. Размещение в workspace

- [ ] Создать приватный удалённый репозиторий отдельным процессом.
- [ ] Перенести утверждённые материалы без дублирования.
- [ ] Добавить repository в superproject как submodule.
- [ ] Настроить регулярный пересмотр документов.
