# Видение продукта

## Назначение

Crimea Travel Platform — независимая цифровая платформа для планирования
поездок по Республике Крым. Это не официальное государственное приложение и не
представляет органы власти.

Первый запуск ориентирован на Крым, но продукт и данные проектируются для
нескольких регионов через иерархию
`Country -> Region -> Locality -> Place`.

## Проблема

Сведения о туристических местах, режиме работы, сезонных ограничениях,
безопасности и доступности распределены по разным источникам и быстро
устаревают. Путешественнику сложно совместить интересы, время, способ
передвижения и реальные условия посещения в один выполнимый маршрут. В
природных и горных локациях проблему усиливают отсутствие точного адреса,
несколько entrances и нестабильный мобильный интернет.

## Целевая аудитория

- самостоятельные путешественники, планирующие поездки по Крыму;
- семьи с детьми;
- автомобилисты и пользователи walking routes;
- путешественники с accessibility needs;
- редакторы лицензированного туристического контента;
- в дальнейшем — посетители других подключённых регионов.

## Пользовательская ценность

Платформа помогает путешественнику:

- находить места по географии, категории, сезону и условиям посещения;
- получать актуальные расписания, сведения о входах и временных закрытиях;
- учитывать safety warnings, accessibility и необходимое equipment;
- выбирать редакционные PreparedRoute;
- генерировать GeneratedRoute по времени, интересам и ограничениям;
- сохранять избранные места и маршруты в своём профиле.

## Основные сценарии

1. Registration, login и управление profile.
2. Выбор region и locality.
3. Просмотр categories, каталога и place card.
4. Просмотр PreparedRoute и route geometry на map.
5. Добавление place в FavoritePlace и route в SavedRoute.
6. Заполнение RouteGenerationRequest.
7. Получение упорядоченных stops, geometry, distance, duration и warnings.
8. Повторное открытие сохранённых данных при нестабильной сети в следующих
   итерациях offline-first.

## Принципы

1. Безопасность и актуальность важнее полноты каталога.
2. Источник и freshness критичных данных должны быть видимы и проверяемы.
3. География и доменная модель не должны замыкаться на одном регионе.
4. Маршрутные рекомендации объяснимы и не скрывают ограничения поставщиков.
5. Внешние routing services подключаются через заменяемый контракт.
6. Пользовательский контент и персональные данные разделяются с редакционными
   и географическими данными.
7. Интерфейс не создаёт впечатление официального государственного сервиса.

## Границы первого MVP

- каталог мест и категорий;
- карточка места с entrances, schedules, closure, seasonality и safety data;
- PreparedRoute с упорядоченными остановками;
- route builder через контракт `RoutingProvider`; на foundation-этапе вместо
  внешнего provider используется deterministic stub;
- регистрация, профиль, FavoritePlace и SavedRoute;
- серверное API и редакционно управляемые данные;
- mobile client для основных пользовательских сценариев.

Текущая работа создаёт только foundation и не реализует перечисленные функции.

## Не входит в MVP

- production-развёртывание и круглосуточная эксплуатация;
- платежи и бронирование;
- социальная сеть и пользовательские отзывы;
- собственный routing engine;
- общественный транспорт и multimodal routing;
- полноценная offline navigation и background synchronization;
- push notifications об изменениях;
- пользовательские публикации и marketplace;
- production Admin Application;
- автоматическое распространение на новые регионы без редакционной подготовки;
- заявления о гарантированной безопасности или официальном статусе данных.

## Критерии успеха MVP

- пользователь находит подходящее место и понимает условия посещения;
- маршрут строится предсказуемо даже без внешнего routing provider;
- данные имеют источник, дату проверки и понятный статус свежести;
- новый регион добавляется без изменения core domain model;
- mobile и backend развиваются независимо при стабильных API contracts.
